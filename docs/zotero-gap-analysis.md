# Lume 对标 Zotero 的功能差距分析

> 首次版本：2026-03-07 | **最后更新：2026-03-09**
> 目的：基于当前仓库现状，梳理 Lume 作为"学术论文阅读软件"要对标 Zotero 仍缺少的关键能力。
> 说明：本文档只做产品与功能分析，不包含代码实现。
> **标注说明：** ✅ 已完成 | 🔧 部分实现 | ❌ 尚未开始

## 一、当前版本已经具备的能力

### 原有基础能力（v0.1 骨架）

**文献库基础**
- 本地论文库目录（`My Library`）
- 文件夹 / 子文件夹管理
- PDF 导入、删除、重命名
- 根目录查看全部论文，子目录查看当前目录论文
- 左侧文件夹树 + 中间论文列表

**PDF 阅读**
- 多标签切换、页面渲染与懒加载
- 页码跳转、放大 / 缩小、适应宽度 / 适应高度
- 文本层加载与文本选择

**基础批注**
- 手写绘制、荧光笔标注、文本批注
- 批注本地保存（JSON sidecar）

## 三、P0：下一阶段必须补齐的核心能力

### 1. 论文级笔记系统缺失 🔴
**当前现状：** 有页面级文字批注，但无法围绕一篇论文建立独立笔记。
**Zotero 的能力：** 每篇文献可以附挂结构化笔记，批注可沉淀为知识卡片。

**缺少内容：**
- 论文级笔记面板（独立于 PDF 页面批注之外）
- Markdown 格式笔记编辑器（支持标题、列表、代码块）
- 从现有高亮批注一键生成笔记段落
- 笔记内引用原文（引用页码 + 文字片段）
- 笔记数据持久化至 SQLite（`notes` 表）
- 笔记内容纳入全局搜索索引

**为什么重要：**
学术阅读的终点不是"看完 PDF"，而是"形成可积累的知识卡片"。没有笔记系统，Lume 只是个阅读工具，而非研究工作台。

---

### 2. 文献条目模型仍不完整 🔧
**当前现状：** 已有完整 bibliographic 字段，但条目类型固定，一个条目只能挂一个 PDF 附件。
**Zotero 的能力：** 条目支持多种类型，一个条目可挂多个异构附件。

**缺少内容：**
- 条目类型选择器（期刊论文 / 会议论文 / 书籍 / 书籍章节 / 学位论文 / 网页等）
- 根据类型动态显示相关字段（如 conference、edition、chapter 等）
- 一个条目挂多个附件（PDF + 补充材料 + 网页快照）
- 非 PDF 条目（纯元数据条目，无附件）

**为什么重要：**
条目类型影响引用格式输出的正确性（@article vs @book vs @inproceedings），也是 BibTeX 导入/导出的基础。

---

### 3. 批注管理视图缺失 🔴
**当前现状：** 批注已能写，但缺少"管理层"——无法一览和跳转。
**Zotero 的能力：** 所有批注可在侧边栏列表中浏览并跳转。

**缺少内容：**
- PDF 阅读器内右侧批注列表面板
- 按页码排序展示所有批注（手写 / 高亮 / 文字）
- 点击批注条目跳转到对应页面
- 批注分类视图（按类型过滤）
- 批注内容关键词搜索
- 批注导出为 Markdown 文件

**为什么重要：**
没有批注列表，用户读完论文后无法快速回顾和整理批注内容。

## 四、P1：强烈建议补齐的高价值能力

### 4. 阅读器基础能力仍偏弱 🔧
**当前现状：** 渲染、翻页、批注可用，但缺少现代 PDF 阅读器的标准功能。

**当前已有：** 渲染、懒加载、翻页、缩放、文本层、多种批注工具

**仍缺少：**
- **PDF 内文本搜索**（关键词高亮 + 上下跳转）— 后端 `get_text_rects` 已具备基础能力
- 目录 / Outline / 书签侧边栏
- 缩略图导航侧栏
- 快捷键体系（← → 翻页、Ctrl+F 搜索、Ctrl+= 缩放等）
- 历史阅读位置记忆（关闭后重新打开恢复位置）
- 最近打开文献列表
- 橡皮擦 / 撤销 / 重做批注操作

**为什么重要：**
阅读器体验是用户留存的底盘，这些是用户每天使用都会感知到的摩擦点。

---

### 5. 文献导入方式太少 🔧
**当前现状：** 已支持 DOI / arXiv 元数据联网补全，但需手动触发，主入口仍是本地文件选择器。

**当前已有：** 本地 PDF 导入、DOI/arXiv 联网抓取（元数据补全触发）

**仍缺少：**
- "通过 DOI / arXiv ID 新建条目"入口（先建 item，再关联 PDF 或不关联）
- 拖入 PDF 后**自动**联网识别并填写元数据（当前需手动触发）
- BibTeX / RIS / CSL JSON 文件**导入**（批量建库）
- 批量导入时的进度反馈与错误报告
- 文件夹监听（指定目录，新增 PDF 自动入库）

**为什么重要：**
用户的真实习惯是"随手把论文丢进来"，而不是"逐个填写元数据"。自动识别可大幅降低入库摩擦。

---

### 6. 重复文献识别与合并缺失 ❌
**当前现状：** 无任何去重机制。

**缺少内容：**
- 导入时检测相同 DOI / 文件哈希的重复项
- 重复项视图（高亮显示冲突字段）
- 合并：保留最优 PDF + 最优元数据
- 去重报告

---

### 7. 文献组织方式仍是纯文件夹思维 ❌
**当前现状：** 完全依赖物理目录树，没有跨文件夹的逻辑聚合能力。

**缺少内容：**
- Smart Collections / Saved Searches（按条件自动聚合）
- 系统虚拟视图：未读 / 最近添加 / 最近打开 / 已加星标
- 同一条目出现在多个逻辑集合中（不复制文件）
- 按年份 / 作者 / 期刊的虚拟分组视图

---

### 8. 元数据来源与校正能力不足 🔧
**当前现状：** 已支持 Crossref + arXiv 双源，但无多源回退 UI 和冲突确认流程。

**仍缺少：**
- Semantic Scholar / OpenAlex / PMID / ISBN 等额外来源
- 批量补全元数据（对整个文件夹批量触发联网抓取）
- 元数据冲突对比 UI（显示"本地 vs API 返回"差异，让用户选择）
- 缺字段检测（在文献列表中标记元数据不完整的条目）

---

## 五、P2：做强产品竞争力的重要能力

### 9. BibTeX / RIS 文件导入 ❌
**当前现状：** 只能导出，无法从外部引用管理器批量导入。

**缺少内容：**
- 解析 `.bib` 文件，批量创建 items（含元数据）
- 解析 `.ris` 文件
- 从 Zotero 导出的 Better BibTeX 格式兼容

**为什么重要：**
这是 Lume 能否承接 Zotero 用户迁移的关键能力。

---

### 10. 同步能力缺失 ❌
**当前现状：** 完全单机本地应用。

**缺少内容：**
- iCloud Drive / 自定义路径数据库同步
- 批注和笔记跨设备同步
- 冲突解决策略

---

### 11. 写作集成能力缺失 ❌

**缺少内容：**
- 全局 Cite Picker（快捷键唤起，搜索并复制引用到写作工具）
- Obsidian / Logseq 格式笔记导出
- Overleaf / LaTeX 工作流（BibTeX 导出已部分覆盖）

---

### 12. 团队协作能力缺失 ❌
- 团队共享文献库
- 共享标签体系
- 评论与讨论流

---

## 六、P3：体验层值得补齐的能力

### 13. 基础桌面体验
- 深色模式
- 全局快捷键注册（系统级，非 PDF 内）
- 窗口 / 标签状态恢复（重启后恢复上次会话）
- 收藏 / 置顶文献
- 回收站（软删除 + 批量恢复）
- 批量操作：多选后批量打标签 / 批量移动 / 批量导出

### 14. 性能与大库体验
- 上万篇论文下的列表虚拟滚动（当前为全量渲染）
- 后台全文索引（SQLite FTS5）
- 元数据异步刷新队列（不阻塞 UI）
- 增量目录扫描（只处理新增/变更文件）

### 15. 扫描件与 OCR
- OCR 识别扫描版 PDF（Tesseract 集成）
- OCR 文本纳入全文搜索

## 八、下一个 Sprint 重点推荐

基于当前进度，第一阶段还剩两项核心能力，建议按以下顺序推进：

**① 论文级笔记系统**（P0 中优先级最高的未完成项）
- MetaPanel 底部新增 Notes 标签页，内嵌 Markdown 编辑器
- SQLite 新增 `notes` 表：`id / item_id / content / created_at / updated_at`
- 笔记内容纳入 `search_library` 全局搜索
- 可从批注内容一键追加到笔记

**② 批注管理视图**（配合已有批注能力，大幅提升回顾效率）
- PDF 阅读器右侧面板展示当前文档所有批注
- 按页码排序，点击条目跳转到对应页

**③ PDF 内文本搜索**（阅读器最高频刚需，后端能力已具备基础）
- 浮动搜索框（Ctrl+F 唤起）
- 利用已有 `get_text_rects` Tauri 命令实现关键词匹配
- 匹配结果高亮层叠 + 上下跳转计数器（1 / 23）

## 九、结论

**2026-03-07 → 2026-03-09 三天内完成：**

> 元数据编辑 ✅ → 全局检索 ✅ → 标签系统 ✅ → 引用导出 ✅

这四项是对标 Zotero 最核心的「文献数据库能力」骨架。Lume 现在已经是一个**真正可用的文献管理器**，而不仅仅是 PDF 阅读器。

**接下来的关键差距集中在：**

> **笔记沉淀 + 批注复用 + PDF 内搜索 + 文献导入生态 + 同步协作**

这几项补齐后，Lume 将全面进入 Zotero 的竞争射程。
